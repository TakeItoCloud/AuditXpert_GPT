## Decisions made
- Build one platform with shared core instead of multiple disconnected tools.
- Use three major modules: Governance/vCISO, Technical Assessment, and AI Explainer.
- Keep remediation out of first release.
- Make PowerShell the default implementation language.
- Treat Azure as a first-class assessment scope, not an afterthought.
- Track otherwise-empty structural folders with placeholder files so the repository skeleton remains durable in source control.

## Known issues
- Exact connector coverage may evolve as Microsoft APIs and permissions change.
- Some Secure Score or Defender data surfaces may require portal/API strategy refinement during implementation.
- Module scaffolds currently expose only placeholder status commands until later phases implement the actual engines.
- The shared core and prerequisite modules are still scaffold-only, so connector readiness logic is self-contained for now.
- The core runtime phases remain incomplete, so the findings engine currently operates independently of shared config and run-context services.
- Core runtime depth, richer packaging options, and broader real-environment validation remain future roadmap items.
- The launcher is currently a UI shell only and still needs a formal configuration contract plus PowerShell binding workflow in a later phase.
- The launcher still needs richer controls, help windows, and explicit execution bindings that consume the new configuration contract.
- The launcher still needs process bridging so UI actions can safely invoke PowerShell entry points.
